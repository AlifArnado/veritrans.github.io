Transaction Charge

Transaction charge allows you to receive payments online. This section will cover the specifications and a brief tutorial of basic transaction charge on VT-Web product and VT-Direct. Both products will be described seperately due to their difference in charging transactions. In VT-Web, the transaction charge function is used to obtain payment URL. In VT-Direct, the function is used for payment.

The explanation of transaction charge is divided into 7 sections:

  1. HTTP(S) POST specification
  2. Example of VT-Web (Credit Card, Mandiri Clickpay, CIMB Clicks, Bank Transfer)
  3. Example of VT-Direct for Credit Card
  4. Example of VT-Direct for Mandiri Clickpay
  5. Example of VT-Direct for CIMB Clicks
  6. Example of VT-Direct for Virtual Account
  7. Example of VT-Direct for e-Pay BRI

There are various parameters in a transaction charge, some parameters need to be configured to activate several features such as: Installment, BIN Promo, one click, and two clicks. Transaction charge features will be described systematically in different sections.

  • Specification
  • VT-Web
  • Credit Card
  • Mandiri Clickpay
  • CIMB Clicks
  • Virtual Account
  • e-Pay BRI


HTTP(S) POST Specification

Description Do charge transaction
Endpoint BASE_URL/charge
HTTP Method POST
HTTP Header Accept: application/json
Content-Type: application/json
Authorization: Basic Base64(server_key:)


JSON Request

Description of JSON request transaction charge:

JSON Attribute Data Type Required Description
payment_type String Yes used payment method.
VT-Direct: "credit_card", "mandiri_clickpay", "cimb_clicks", "bank_transfer" or "bri_epay"
VT-WEB: "vtweb"
vtweb JSON Object No VT-Web Configuration
transaction_details JSON Object Yes Transaction details that will be paid to customers
item_details JSON Array No Shopping item details that will be paid by customer
customer_details JSON Object Conditional customer data details for making payment
Note:
VT-Direct: Required
VT-Web: Optional
credit_card JSON Object Conditional VT-Direct charge details using credit card
Note: Only for VT-Direct that is using credit card
mandiri_clickpay JSON Object Conditional VT-Direct charge details using Mandiri Clickpay
Note: Only for VT-Direct that is using Mandiri Clickpay
cimb_clicks JSON Object Conditional VT-Direct charge details using CIMB Clicks
Note: Only for VT-Direct that is using CIMB Clicks
bank_transfer JSON Object Conditional VT-Direct charge details using Virtual Account
Note: Only for VT-Direct that is using Virtual Account


request.vtweb object description.

JSON Attribute Data Type Required Description
enabled_payments JSON Array No List of payment method that can be used by customer
Array element string that is allowed: "credit_card", "cimb_clicks", "mandiri_clickpay", "bank_transfer" and "bri_epay".
Note: If this attribute is unused, all payment methods registered to merchant will automatically used.
credit_card_3d_secure Boolean No Status to indicate whether 3D secure is used or not
payment_options JSON Object No Payment features that can be used on VT-Web. Example: installment.
credit_card_bins JSON Array No List of credit card's BIN (Bank Identification Number) that can be used on VT-Web. All BIN can have 1 to 8 digits.


request.installment object description.

Note: Required status is only applied if installment object is used in payment_options

Attribute Data Type Required Description
required Boolean Yes Status to indicate whether installment is required or not
installment_terms JSON Object Yes Bank options and installment period


request.installment.installment_terms Object Description.

Note: At least one bank and one installment period have to exist on installment_terms object

Attribute Data Type Required Description
bni JSON Array No List of installment periods using BNI Bank
Note: In Sandbox environment, valid array elements are 6 and 12. If array is empty, all valid installment period will be shown.
mandiri JSON Array No List of Installment term using Mandiri bank
Note: In Sandbox environment, valid array elements are 6 and 12. If array is empty, all valid installment period will be shown.


request.payment_options object description:

Attribute Data Type Required Description
installment JSON Object No Detail of Installment Feature in VT-Web


request.credit_card object description.

Note: required status only applied if credit card object is used in JSON request

JSON Attribute Data Type Required Description
token_id String Yes Token ID represents customer's credit card information
bank String Yes Acquiring bank. Valid values for
Production: "bni", "mandiri", or "cimb"
Sandbox: "bni" or "mandiri"
installment_term Integer No Installment Term. Valid value for Sandbox: 6 or 12
bins JSON Array No List of BINs that are allowed for transaction. All BIN can have 1 to 8 digit(s)
type String No Used on pre-authorization feature. Valid value: "authorize"
save_token_id Boolean No Used on 'one click' or 'two click' feature. Status to indicate whether token_id can be used for the next transaction


request.mandiri_clickpay Object Description.

Note: required Status only applies if mandiri_clickpay object is used in JSON request

input1 String Yes last 10 digit card_number

input3 String Yes 5-digits random number which will be given to the customer
token String Yes Number given by customers machine token after input1, input2, and input3 are entered

Note: required status only applies if cimb_clicks object is used in JSON request

Error on Transaction Charge

On several occasions, VT-Direct transaction charge might be denied. Veritrans will send back the error message when this happens. The reason for the failure can be checked from the status code on the error message. Typical status codes that usually occur are:

Error Code Description
400 Validation error Validation Error, merchant send bad request data example; validation error, invalid transaction type, invalid credit card format, etc.
401 Access Denied Invalid client key or server key
402 Access Denied Merchant doesn't have access for this payment type
406 Duplicate Order ID Order ID has been used before or transaction has been paid.
410 Account Inactive Merchant account is deactivated or not active for a long period
411 Token error Token id is missing, invalid, or timed out

Complete list of status codes can be checked on API Reference - Status Code.


Structure and Example of JSON Response VT-Direct using Credit Card

If transaction charge has no errors, Veritrans server will send a JSON response. The Structure of JSON Response VT-Direct using credit card is as followed:

JSON Attribute Data Type Description
transaction_id String Transaction ID given by Veritrans
order_id String Order Id given by merchant
gross_amount String total transaction
payment_type String Transaction payment method
transaction_time String Timestamp of transaction
transaction_status String Transaction status. After charge credit card transaction, the possible values are
capture : Transaction is accepted by the bank. If fraud_status value is challenge, merchant have to accept the transaction manually.
deny: transaction is denied by the bank or FDS.
authorize: Credit card is valid for transaction (pre-authorization)
fraud_status String Detection result by Fraud Detection System (FDS). Possible Value:
accept : Approved by FDS.
deny: Denied by FDS. Transaction automatically failed.
challenge: Questioned by FDS. If transaction_status is capture, merchant should accept the transaction manually
masked_card String first 6-digit and last 4-digit of customer's credit card number
status_code String Status code of transaction charge result. Complete list of status codes can be seen here
status_message String Description of transaction charge result
approval_code String Approval code. It can be used for refund. This does not exist on denied transaction

Error on Transaction Charge

On several occassions, virtual account transaction charge may be denied, Veritrans will send back an error message when this happens. The reason of the failure can be checked through the status code on the error message. Status code that usually occur are:

Error Code Description
400 Validation error Validation Error, merchant send bad request data example; validation error, invalid transaction type, invalid credit card format, etc.
401 Access Denied Invalid client key or server key
402 Access Denied Merchant doesn't have access for this payment type
406 Duplicate Order ID Order ID has been used before or transaction has been paid.
410 Account Inactive Merchant account is deactivated or not active for a long period

Complete list of status codes can be checked on API Reference - Status Code.
