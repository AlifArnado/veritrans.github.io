Other Features

Currently, VT-Direct has additional features for credit card payment method, which are:

  1. Pre-Authorization & Capture
  2. One Click
  3. Two Clicks

The following is tutorial to utilise those features as needed.

  • Pre-Authorization & Capture
  • One Click
  • Two Clicks

Pre-authorization & Capture

Pre-authorization is used to check whether a credit card is able to perform the transaction with a specified amount. New transaction only occurs after capture process. Only VT-Direct using credit cards is allowed to use this feature. Further explanation can be checked on VT-Direct - Pre-Authorization & Capture.

Pre-authorization

In VT-Direct, you have to obtain a authorization token ID first. As to be concerned, the request token parameter for pre-authorization is different with normal transaction. You have to add the "type" attribute on request token parameter.

After pre-authorization token is obtained, you need to add "type" attribute in "credit_card" when charging is being done. Pre-authorization function specification is similar to credit card transaction charge.

{
    // Required
    "payment_type" : "credit_card",
    // Required
    "credit_card" : {
        "bank" : "bni",
        "token_id": "< your token ID >",
        "type" : "authorize"
    },
    // Required
    "transaction_details" : {
    "order_id" : "A27550",
    "gross_amount" : 145000
  },
    // Required
    "customer_details" : {
      "first_name" : "Andri",
      "last_name" : "Litani", // Optional
      "email" : "andri@litani.com",
      "phone" : "081122334455",
        // Optional
      "billing_address" : {
          "first_name" : "Andri",
          "last_name" : "Litani",
          "address" : "Mangga 20",
          "city" : "Jakarta",
          "postal_code" : "16602",
          "phone" : "081122334455",
          "country_code" : "IDN"
        },
        // Optional
      "shipping_address" : {
          "first_name" : "Obet",
          "last_name" : "Supriadi",
          "address" : "Manggis 90",
          "city" : "Jakarta",
          "postal_code" : "16601",
          "phone" : "08113366345",
          "country_code" : "IDN"
        }
      },
    // Optional
    "item_details" : [
    {
        "id" : "a1",
        "price" : 50000,
        "quantity" : 2,
        "name" : "Apel"
    },
    {
        "id" : "a2",
        "price" : 45000,
        "quantity" : 1,
        "name" : "Jeruk"
    }

    ]
}

Pre-authorization JSON response structure is similar to credit card transaction charge. When the process is successful, "transaction_status" value will be "authorize". Take note that, the transaction has not happened yet, the transaction occurs only when the capture function is performed. The following is example for pre-authorization JSON response :

{
    "status_code": "200",
    "status_message": "OK, Credit Card 3D Secure transaction is successful",
    "transaction_id": "1ac1a089d-a587-40f1-a936-a7770667d6dd",
    "order_id": "A27550",
    "payment_type": "credit_card",
    "transaction_time": "2014-08-25 10:20:54",
    "transaction_status": "authorize",
    "fraud_status": "accept",
    "masked_card": "481111-1114",
    "approval_code": "1408936854718",
    "gross_amount": "145000.00"
}

Capture

After a successful pre-authorization process, transaction charge can be done using capture function. You need to record pre-authorization "transaction_id" and use it as one of the attributes of JSON request capture.

Spesification

Description Transaction charge which have passed pre-authorization
Endpoint BASE_URL/capture
HTTP Method POST
HTTP Header Accept: application/json
Content-Type: application/json
Authentication: Basic Base64(server_key:)

JSON Request

The following is description for JSON Request capture transaction:

JSON Attribute Data Type Required note
transaction_id String Yes Transaction ID which has passed pre-authorization
gross_amount Long Ya The value of transactions which will be paid by the customer.
Note: Should not exceed the value of current pre-authorization transaction

The following is example of JSON request capture :

{
    "transaction_id" : "1ac1a089d-a587-40f1-a936-a7770667d6dd",
    "gross_amount" : 55000
}

Take note that the value of transactions in above example is smaller than the value during pre-authorization transaction.

JSON Response

Description of JSON response is similar to normal transaction charge. The following is example of successful capture response :

{
    "status_code": "200",
    "status_message": "OK, Credit Card capture transaction is successful",
    "transaction_id": "1ac1a089d-a587-40f1-a936-a7770667d6dd",
    "order_id": "A27550",
    "payment_type": "credit_card",
    "transaction_time": "2014-08-25 10:20:54",
    "transaction_status": "capture",
    "fraud_status": "accept",
    "masked_card": "481111-1114",
    "approval_code": "1408937217061",
    "gross_amount": "55000.00"
}

One Click

One click feature allows you to use one token on several transactions. Token represents the credit card number, expiry date, CVV, and 3D Secure authentication of the customer. This will ease the transaction because the credit card information only need to be filled once.

Do take note that token will be activated only after the initial transaction is performed. To use the one click feature, 3D Secure feature has to be used.

In general, there are two steps required to use one click feature:

  1. Initial Transaction
    • Customer fills the credit card number, CVV, expiry date.
    • Customer performs 3D Secure authentication.
    • Merchant receives token.
    • Merchant uses token for transaction charge.
    • When transaction is successful, merchant saves customer’s token in the database. We name this token as saved_token.
  2. Following Transaction
    • Merchant retrieves customer saved_token from database.
    • Merchant uses saved_token to transaction charge.

One click transaction charge specification is similar to the credit card transaction charge. You only need to add the save_token_id attribute on credit_card JSON request

Two Clicks

Similar to one click feature, two clicks allows your customers to do easier and faster transactions. After the first transaction succeed, you can save the tokens for the next transactions.

Two clicks token represent the number and expiry date of your customer's credit card. Your customer only need to fill out the CVV and 3D Secure authentication for the subsequent transactions. In contrast with one click feature, two clicks feature can be used on both regular and 3D Secure credit card transactions.

In general, there are two steps required to use two clicks feature:

  1. Initial Transaction
    • Customer fills the credit card number, CVV, expiry date.
    • Customer performs 3D Secure authentication.
    • Merchant receives token.
    • Merchant uses token to charge transaction.
    • When transaction is successful, merchant save customer’s token in the database. We name this token as saved_token.
  2. Following Transaction
    • Merchant retrieves customer saved_token from database.
    • Customer fills in CVV.
    • Perform 3D Secure authentication if required
    • Merchant uses saved_token and CVV to acquired new token
    • Merchant uses new token to charge transaction

First transaction charge for Two Clicks has similar format to the first transaction charge on One Click. During the first transaction charge, "save_token_id" have to be added on "credit_card" part of JSON request. Thus, the token will be stored for the future payments.

Next Transaction Charge

For the next transactions, we use "saved_token_id", CVV, and 3DS authentication (if enabled) that is entered by the customer to obtain the new token. two_click attributes also need to be added when new request token is performed. The new token will be used for the subsequent transactions. The new token will be filled automatically into <input id="token-id" name="token-id" type="hidden" /> by veritrans.min.js.

Field number and expiry date field can be omitted because the customer only require to fill the CVV and perform 3D Secure authentication. Using the same example in charge transaction, the following is the new request token using the saved_token and CVV entered by customer:
